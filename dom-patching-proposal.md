# Proposal: DOM Patching for Markdown Preview to Prevent Flickering

## 1. Problem Analysis
The preview pane currently flickers when the Python `markdown-it` library generates a new HTML string and the frontend updates the view. This happens because replacing the `.innerHTML` of a container completely destroys the existing DOM tree and reconstructs it from scratch. This causes:
- Complete loss of scroll position.
- Flashing/flickering as the browser recalculates layout.
- Re-fetching of external resources (like images or Mermaid diagrams).
- Resetting of any interactive state within the preview.

To fix this, we need a way to **morph** the existing DOM to match the new HTML tree, touching only the nodes and attributes that have actually changed.

## 2. Debian Package Ecosystem Evaluation
Per the architectural mandate ([SPEC §2.2](SPEC.md#22-runtime-dependencies-debian-packages)), any external library must be available as a Debian package (e.g., `libjs-*`). I have extensively searched the Debian repositories for DOM diffing, morphing, and Virtual DOM libraries.

### Findings:
- ❌ **Dedicated Morphing Libraries:** Industry-standard DOM patching libraries such as `morphdom`, `idiomorph`, `nanomorph`, and `setdom` are **not currently packaged** in Debian. 
- ⚠️ **Virtual DOM Frameworks:** Packages like `libjs-vue`, `node-react`, and `node-preact` *are* available. However, they are fundamentally incompatible with your architecture. They expect to own the DOM via their own state and render functions. To use them for diffing an arbitrary HTML string (generated by Python), you would need to parse that string into a Virtual DOM on the client-side, which requires additional libraries (like `html-to-react` or `html-to-vdom`) that are also not in Debian.
- ⚠️ **HTMX (`libjs-htmx`):** HTMX is available in Debian. However, its native swap mechanism replaces `innerHTML`, which still causes flickering. To get morphing behavior, HTMX relies on the `idiomorph` extension, which is not included in the Debian package.
- ❌ **Text Diffing:** Packages like `node-diff` (jsdiff) are available but perform generic string diffing. Diffing HTML strings and applying partial string updates to `innerHTML` is highly prone to producing malformed DOM structures and XSS vulnerabilities.

**Conclusion:** There is currently no `libjs-*` framework in Debian perfectly suited for morphing server-rendered HTML strings out of the box.

## 3. Recommended Strategy: Vanilla JS DOMWalker (Zero Dependency)

Given the strict Debian packaging requirement, the most architecturally sound approach is to implement a lightweight, native JavaScript DOM diffing function. Since the HTML is generated from Markdown, the DOM structures are relatively predictable (mostly text nodes, spans, divs, and images).

### How it works:
1. Receive the new HTML string from Python.
2. Parse it into an offline DOM tree using the browser's native `DOMParser`.
3. Recursively walk both the live DOM and the offline DOM.
4. Update only the differing text nodes, attributes, and append/remove elements where necessary.

### Advantages:
- **Zero Dependencies:** Fully adheres to the project's strict dependency rules without waiting for a new Debian package to be minted.
- **High Performance:** `DOMParser` is native C++ in WebKitGTK. A simple recursive tree-walker is extremely fast for standard Markdown documents.
- **No Build Step:** It’s just plain JavaScript that can be loaded directly in the WebKitGTK view.

### Proof-of-Concept Implementation:
Here is a concise implementation that handles the core morphing logic:

```javascript
function updatePreview(newHtmlString) {
    const parser = new DOMParser();
    const newDocument = parser.parseFromString(newHtmlString, 'text/html');
    const newContent = newDocument.body;
    const oldContent = document.getElementById('preview-container');

    morphNode(oldContent, newContent);
}

function morphNode(oldNode, newNode) {
    // 1. If nodes are different types or different tags, replace entirely
    if (oldNode.nodeType !== newNode.nodeType || oldNode.nodeName !== newNode.nodeName) {
        oldNode.replaceWith(newNode.cloneNode(true));
        return;
    }

    // 2. If it's a text node, update the text if it changed
    if (oldNode.nodeType === Node.TEXT_NODE) {
        if (oldNode.nodeValue !== newNode.nodeValue) {
            oldNode.nodeValue = newNode.nodeValue;
        }
        return;
    }

    // 3. Update attributes on element nodes
    if (oldNode.nodeType === Node.ELEMENT_NODE) {
        // Remove old attributes not present in the new node
        for (const attr of Array.from(oldNode.attributes)) {
            if (!newNode.hasAttribute(attr.name)) {
                oldNode.removeAttribute(attr.name);
            }
        }
        // Set new/updated attributes
        for (const attr of Array.from(newNode.attributes)) {
            if (oldNode.getAttribute(attr.name) !== attr.value) {
                oldNode.setAttribute(attr.name, attr.value);
            }
        }
    }

    // 4. Diff child nodes
    const oldChildren = Array.from(oldNode.childNodes);
    const newChildren = Array.from(newNode.childNodes);
    const max = Math.max(oldChildren.length, newChildren.length);

    for (let i = 0; i < max; i++) {
        if (!oldChildren[i]) {
            // New node added
            oldNode.appendChild(newChildren[i].cloneNode(true));
        } else if (!newChildren[i]) {
            // Old node removed
            oldNode.removeChild(oldChildren[i]);
        } else {
            // Recursively morph existing node
            morphNode(oldChildren[i], newChildren[i]);
        }
    }
}
```

## 4. Fallback Alternative: Vendor `morphdom`
If the simple vanilla JS walker proves insufficient for complex Mermaid diagrams or nested transclusions, the fallback option is to vendor the `morphdom` library directly into the source tree (as a single `morphdom.js` file in the UI assets directory). 

While [SPEC §2.2](SPEC.md#22-runtime-dependencies-debian-packages) mandates Debian packages for dependencies, exceptions are occasionally made for single-file, zero-dependency frontend assets (similar to how Mermaid might be cached). If you choose this route, we would file an Issue to track the long-term intent of packaging `morphdom` into Debian via an `ITP` (Intent to Package).

### Addendum: Evaluation of Mithril (`node-mithril`)
I also evaluated Mithril as a potential solution, as `node-mithril` is available in the Debian repositories. However, it suffers from the exact same structural mismatch as Vue or React:

- **How Mithril works:** Mithril is a Virtual DOM (VDOM) framework. It expects you to define the UI tree using its JavaScript `m()` function (e.g., `m("div", m("p", "text"))`). When state changes, it compares the new VDOM against the old VDOM and patches the real DOM.
- **The String Problem:** Our pipeline outputs a raw HTML string from Python's `markdown-it`. To inject an HTML string in Mithril, you must use `m.trust(htmlString)`.
- **Why it still flickers:** When Mithril encounters `m.trust()`, it does **not** parse the HTML into VDOM nodes. It treats the string as an opaque blob and simply assigns it via `.innerHTML` (or `insertAdjacentHTML`). This destroys and rebuilds the entire DOM subtree, resulting in the exact same flickering, resource re-fetching, and state loss that we are trying to fix.
- **Missing Link:** To get diffing, we would need a client-side parser to convert the HTML string into a tree of `m()` calls *before* handing it to Mithril (e.g., an `html-to-vdom` library). Such libraries are not packaged in Debian.
- **Version Note:** The `node-mithril` package in Debian is also very outdated (version 1.1.7, released around 2017).

Thus, Mithril cannot natively morph a raw HTML string and does not solve the underlying flickering issue.
